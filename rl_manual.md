# FAFNIR RL AI マニュアル & アルゴリズム解説

本ドキュメントは、FAFNIR ONLINEのゲーム環境で動作する強化学習AI（RL AI）の仕組みと、学習・実行方法をまとめたものです。

---

## 1. 使い方マニュアル (Usage)

コマンドプロンプトやPowerShellを開き、仮想環境を有効化してから実行してください。

### 1.1. 実戦 (Deployment)
学習したAIを実際のサーバーに接続し、自分（人間）や他のAIと対戦させます。

**① サーバーの起動**
```bash
venv\Scripts\python.exe -m uvicorn server:socket_app --host 0.0.0.0 --port 8765
```

**② RLボット（AI）の接続**
別のターミナルを開いてAIを参加させます。
```bash
venv\Scripts\python.exe clients/rl_bot.py --model rl/output/fafnir_final --name RLBot --room room1
```
**【rl_bot.py の主要なオプション】**
| オプション | デフォルト | 説明 |
|---|---|---|
| `--model` | (必須) | 使用する学習済みモデルのパス。 |
| `--name` | `RLBot` | サーバーに接続する際のプレイヤー名。 |
| `--room` | `room1` | 接続するルームID。 |
| `--think-delay`| `0.05` | AIが行動を決定する際の思考遅延（秒）。人間が観戦しやすいように意図的に設定します。 |

**③ 人間の接続（対戦用GUI）**
人間はターミナルではなく、ブラウザベースのグラフィカルなWeb GUIからプレイするのがおすすめです。
`clients/web_gui/index.html` のファイルをダブルクリックしてブラウザ（ChromeやEdgeなど）で開きます。
画面上の「Join Game」ボタンを押すだけで自動的にサーバーに入室し、リッチなグラフィックとマウス操作でAIと対戦が可能です。

※従来のCUIで遊びたい場合は以下のコマンドを使用します:
```bash
venv\Scripts\python.exe clients/human_cli.py --name Human --room room1
```

### 1.2. 学習の実行 (Training)
本番用の学習を行うコマンドです。GPU（RTX3050等）を活かすため `--device cuda` を指定し、並列環境数（`--n-envs`）を増やして高速化します。

```bash
venv\Scripts\python.exe rl/train.py --total-steps 1000000 --device cuda --n-envs 8 --update-freq 50000 --save-freq 250000

venv\Scripts\python.exe rl/train.py --total-steps 1000000 --device cuda --n-envs 8 --update-freq 50000 --save-freq 250000 --resume rl/output/fafnir_final.zip
```
**【train.py の主要なオプション】**
| オプション | デフォルト | 説明 |
|---|---|---|
| `--total-steps` | `500000` | AIが行動する合計回数（ステップ数）。多いほど学習が進みますが時間がかかります。 |
| `--n-envs` | `4` | 並列環境数。同時にいくつのゲームを進行させるか。CPUのコア数に合わせる（4〜8など）と大幅に高速化します。 |
| `--device` | `auto` | 学習に使用するハードウェア。`cuda` を指定するとGPUを使用して高速化します。 |
| `--resume` | なし | 途中で止めた学習を再開するときに、保存された `.zip` ファイルのパスを指定します。 |
| `--update-freq` | `20000` | 自己対戦において、対戦相手を「最新の自分」に更新するまでのステップ数。 |
| `--save-freq` | `50000` | モデルを `.zip` として物理保存する間隔。 |
| `--score-to-win` | `30` | 何点先取でゲーム終了とするか。学習時は短時間で勝敗を経験させるため低めに設定しています。 |

- 学習の進捗は `rl/output` フォルダ内に保存されます。
- **TensorBoardでの進捗確認:** 別のターミナルを開き `venv\Scripts\python.exe -m tensorboard.main --logdir rl/output/logs` で学習グラフをブラウザで確認できます。

### 1.3. 評価 (Evaluation)
学習したモデルが単純なロジック（ランダム、常に1枚出す貪欲、2〜4枚出すアグレッシブ）に対してどの程度勝てるかをテストします。

```bash
venv\Scripts\python.exe rl/evaluate.py --model rl/output/fafnir_final --games 500
```

## 2. 使用アルゴリズムと学習手法

### 2.1. Maskable PPO (Proximal Policy Optimization)
本AIのコアアルゴリズムには、**Maskable PPO** を採用しています。
- **PPO:** 現在最も広く使われている強化学習アルゴリズムの一つです。学習の安定性が高く、ハイパーパラメータの調整が比較的容易という特徴があります。
- **Masking (行動マスク):** FAFNIRでは「オファーに出ている色と同じ色は入札できない」「手持ちの枚数以上は入札できない」といった厳密なルールがあります。通常のPPOでは無効な行動を選んだ際にペナルティを与えて学習させますが、学習が非効率になります。`sb3-contrib` の `MaskablePPO` を使うことで、ネットワークの出力段階で「現在選択不可能な行動」の確率を強制的に0にし、有効な行動の中だけで学習を最適化できます。

### 2.2. 自己対戦 (Self-Play) とカリキュラム学習
ゲーム理論に基づく不完全情報ゲームにおいて、AIを強くするための標準的なアプローチです。
- AIは最初ランダムに動く相手と対戦してゲームの基本ルール（得点の取り方）を学びます。
- 学習が進むと、**定期的に過去の自分自身（チェックポイント）を対戦相手としてロード**します。
- これにより、「過去の自分を倒すための戦略」を次々と発見し、単調な相手に過剰適合（オーバーフィッティング）することを防ぎつつ、螺旋状に強くなっていきます。

### 2.3. Numba による高速シミュレーション (JITコンパイル)
強化学習では数百万回のゲームをプレイする必要があります。既存の `server.py` はSocket.IOの非同期処理や通信を含むため、学習には不向きです。
そのため、ゲームロジックのコア部分を **Numba (`@njit`)** を用いて純粋なNumPy配列の演算として再実装しました（`fast_engine.py`）。これにより、Pythonのオーバーヘッドをバイパスし、C言語に匹敵する速度（通常のPython実装の数十倍）でシミュレーションを回すことが可能になっています。

## 3. 状態空間・行動空間・報酬の設計

### 3.1. 観測空間 (Observation Space)
AIがゲームの状況を把握するための情報（36次元の正規化された実数ベクトル）です。すべて $0.0 \sim 1.0$ の範囲に正規化されています。
- `0-5`: 自分の手札の各色の枚数
- `6-11`: 現在のオファー石の各色の枚数
- `12-17`: ゴミ箱の各色の枚数（6個で上限）
- `18-23`: 相手の確定手札（オークション結果から推測される最低保有数）
- `24`: 相手の不確定な手札の枚数（合計枚数 − 確定枚数）
- `25-30`: 自分の確定手札（相手に知られている自分の最低保有数）
- `31`: 自分のスコア
- `32`: 相手のスコア
- `33`: 袋の残り枚数
- `34`: 自分が管理人(Caretaker)かどうか (0 or 1)
- `35`: 自分の手札の現在の期待値スコア（色の順位に基づく得点計算の予測値）

### 3.2. 行動空間 (Action Space)
AIが出力する行動です。
`MultiDiscrete([11] * 6)` として定義されており、6つの色（gold, red, orange, yellow, green, blue）それぞれについて、「0〜10個のどれを入札するか」を同時に決定します。

### 3.3. 報酬設計 (Reward Design)
AIが学習するためのスコアです。学習を効率化しつつ、最終的な「ゲームの勝利」を目指させるために、3段階の報酬を与えています。
- **ターンの報酬 (小さな道しるべ):** 1回の競り終了時、およびラウンド終了時、「(自分のスコア増加分 - 相手のスコア増加分) × 0.02」を与えます。得点を稼ぐことへの小さなヒントです。
- **タイムオーバー時:** 最大ターン数（`max_turns`）に達した場合、その時点でスコアが相手より高ければ `+0.5`、低ければ `-0.5` を与えます。
- **ゲーム終了時 (究極の目標):** 目標スコア（`score_to_win`）に達してゲームが終了した際、自分が勝てば `+1.0`、負ければ `-1.0` の特大報酬を与え、最終的な勝利を最も重要視させます。

## 4. ファイル構造と関数の役割詳細

### 4.1. `rl/fast_engine.py` (高速ゲームエンジン)
Pythonのオーバーヘッドを無くすため、Numba (`@njit`) を使用してC言語並みの速度でゲームのシミュレーションを行うコア部分です。
- `fast_reset(...)`: ゲームの初期状態（袋の石、手札、オファー等）をランダムに生成し、各種配列をリセットします。
- `fast_step(...)`: プレイヤーと相手の入札（アクション）を受け取り、勝敗判定、石の移動、スコア計算を行い、次のターンに進めます。報酬の計算もここで行われます。
- `_do_round_end(...)`: ゴミ箱が満杯になるか袋が空になった際の、ラウンド終了処理（手札の得点計算と場のリセット）を行います。
- `build_obs(...)`: 現在のゲームの内部状態配列から、ニューラルネットワークに入力するための25次元の「観測（Observation）」ベクトルを構築します。
- `build_mask(...)`: 現在のオファーや手札の枚数を参照し、ルール上選択してはいけない行動をブロックする「マスク」配列を作成します。

### 4.2. `rl/game_env_fast.py` (Gymnasiumラッパー)
`fast_engine.py` の純粋な関数群を、強化学習ライブラリ（Stable-Baselines3）が扱えるオブジェクト指向の環境（`gymnasium.Env`）としてラップします。
- `__init__(...)`: 観測空間（25次元ベクトル）と行動空間（11パターン×6色）を定義し、各種状態変数を初期化します。
- `reset(...)`: 環境を初期化し、最初の観測ベクトルを返します。
- `step(action)`: AIの行動を受け取り、対戦相手の行動を計算した上で、内部の `fast_step` を呼び出します。その後、新しい観測ベクトルと報酬をAIに返します。
- `get_cui_state()`: ターミナルのライブビュー用に、現在のゲーム盤面を人間が読める文字列フォーマットに変換して返します。

### 4.3. `rl/train.py` (学習スクリプト)
モデルの定義と、学習ループを管理します。
- `train(args)`: マルチプロセス環境（`SubprocVecEnv`）を構築し、MaskablePPOモデルを定義して学習を開始します。
- `SelfPlayCallback` クラス: 自己対戦を実現するためのコールバック。定期的に現在の脳を保存し、並列動作している環境の「対戦相手」を過去の自分やランダムAIに書き換えます。
- `LiveViewCallback` クラス: 数千ステップごとに、環境0から現在の盤面状態（CUI文字列）と直近の勝率を取得し、ターミナルに表示します。

### 4.4. `clients/rl_bot.py` (実戦用クライアント)
学習済みのモデルを、オンライン対戦サーバー（`server.py`）に接続するためのSocket.IOクライアントです。
- `state_to_obs(...)` / `state_to_mask(...)`: サーバーから送られてくるJSON形式の盤面データを受け取り、ニューラルネットワークが理解できる25次元ベクトルとマスク配列に変換します。
- `action_to_stones(...)`: ニューラルネットワークが出力した行動（各色の枚数）を、サーバーに送信するための文字列リスト（例: `['red', 'blue', 'blue']`）に変換します。
- `do_submit_bid(...)`: 自分の入札ターンが回ってきた際に、現在の盤面からモデルを使って推論し、最適な入札をサーバーに送信します。

### 4.5. `clients/web_gui/` (人間用 Web GUI)
Socket.IOを介してサーバーに接続し、人間がブラウザ上で直感的にプレイできるリッチUIクライアントです。
- `index.html`: ゲーム画面の骨組み（ログイン画面、手札エリア、ゴミ箱エリア、対戦ログ等）を定義します。
- `style.css`: ガラスのような半透明パネル（グラスモーフィズムデザイン）や、勝者の王冠（👑）アニメーションなど、美しく遊びやすいデザインスタイルを提供します。
- `app.js`: サーバーとの通信ロジックを担当。ゲーム進行状態（フェーズ）に応じた画面の切り替え、クリック操作による入札枚数の増減、詳細なゲームログの自動記録を行います。オファーと同じ色は入札できないようにUI側で物理的にブロックする保護機能も搭載しています。
