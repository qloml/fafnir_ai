# Fafnir rule / server_0424.py bug report

作成日: 2026-05-29

参照:
- `fafnir_rule.txt`
- `server_0424.py`

## 前提

山札終了条件の「山札から引けない場合」は、ユーザー確認により「必要なときに引けなかった場合」と解釈する。
そのため、ターン準備で2個必要な場面に2個引けない場合の終了判定は、大きな相違とは扱わない。

## 重大なバグ / 相違

### 1. 賭けが同時公開ではなく順番制になっている

ルールでは、各プレイヤーが捨てる宝石を同時に公開する。

現在の実装では `current_bidder` によって提出順が制御されており、先に提出したプレイヤーの後にもう一方が提出する。

該当箇所:
- `fafnir_rule.txt`: 賭けの同時公開
- `server_0424.py`: `current_bidder`
- `server_0424.py`: `submit_bid`

影響:
- 後手が先手の提出状況を見てから選べる可能性がある。
- 同時公開を前提にした駆け引きが成立しない。

### 2. 先に提出した bid が全員へ送信される

`submit_bid` 後に `emit_action()` で `bid` の中身を全員へ送っている。
`emit_action()` はプレイヤーと観戦者の全員へ同一 payload を送るため、後手が先手の bid を見られる可能性がある。

該当箇所:
- `server_0424.py`: `emit_action`
- `server_0424.py`: `submit_bid` action

影響:
- 同時公開ルールに反する。
- 順番制と組み合わさると、後手が強く有利になる。

### 3. プレイヤーにも全員の手札が action 経由で送られる

`resolve_before` と `resolve_after` の payload に `hands: [pp.stones[:] for pp in game.players]` が含まれている。
`emit_action()` は全員へ送信するため、観戦者だけでなくプレイヤーにも相手の手札が送られる。

該当箇所:
- `server_0424.py`: `emit_action`
- `server_0424.py`: `resolve_before`
- `server_0424.py`: `resolve_after`

影響:
- ルールの「手札は基本的に自分のみ見られる、他プレイヤーに非公開」に反する。
- クライアントが表示していなくても、通信データ上は見えてしまう。

### 4. action_log 経由でも全手札情報が残る可能性がある

`emit_action()` は payload を `room.action_log` に保存し、`build_state()` は `action_log` を state に含めている。
全手札入りの `resolve_before` / `resolve_after` がログに残るため、後続の state 更新でもプレイヤーへ相手の手札が渡る可能性がある。

該当箇所:
- `server_0424.py`: `emit_action`
- `server_0424.py`: `build_state`

影響:
- 一度漏れた非公開情報がログとして残り続ける。

### 5. ラウンド終了時、得点計算対象の手札を確認する前に再配布している

ルールでは、ラウンド終了時に各プレイヤーが自分の手札を公開して得点計算する。

現在の実装では `start_round_end()` 内で得点計算後すぐに `reset_all_stones_into_bag_and_redeal()` を呼んでいる。
そのため、`ROUND_END` phase に入った時点ではすでに次ラウンド用の手札・捨て札・ファフニルへ更新されている。

該当箇所:
- `server_0424.py`: `start_round_end`
- `server_0424.py`: `reset_all_stones_into_bag_and_redeal`

影響:
- 得点計算時の手札を画面で確認できない可能性が高い。
- `round_end_info` に得点結果はあるが、元の手札内訳が保存されていない。

## 得点計算のバグ

### 6. 同色5個以上のとき、マイナス点まで無効化している

ルールでは、同色を5個以上持っている場合、そのプレイヤーはプラスの得点を受け取れない。
ただし、マイナスの得点は加算される。

現在の実装では `cnt >= 5` の時点で `continue` しているため、その色が3番目以降の `-1点` 色だった場合でもマイナス点が加算されない。

該当箇所:
- `fafnir_rule.txt`: 得点計算
- `server_0424.py`: `compute_hand_score_for_player`

影響:
- 5個以上持っている低順位色のペナルティが消える。
- 得点がルールより高くなる。

## ルールと異なる可能性がある挙動

### 7. 両者0個公開時に両者 -1 点している

ルールでは、互いに0個を公開した場合、ファフニル上の宝石を捨て札に置くとだけ書かれている。

現在の実装では、両者に `-1` 点してからファフニル上の宝石を捨て札に置く。

該当箇所:
- `fafnir_rule.txt`: 賭け
- `server_0424.py`: `resolve_auction`

影響:
- ルールにない減点が発生する。

確認事項:
- これは意図的な追加ルールか。
- 意図的なら `fafnir_rule.txt` に明記する。
- 意図しないなら実装から削除する。

## 軽微な問題

### 8. 勝利理由の文字列が `SCORE_40` になっている

勝利条件自体は `SCORE_TO_WIN = 1000` で正しい。
ただし、`check_game_end()` が返す reason が `SCORE_40` になっている。

該当箇所:
- `server_0424.py`: `check_game_end`

影響:
- ログや UI 表示で混乱する。

## 不明点

### 1. 観戦者は全手札を見てよいか

`spectator_full` では観戦者に全手札を見せる設計になっている。
これ自体が仕様なら問題ない。

ただし、現在は観戦者だけでなくプレイヤーにも action 経由で全手札が渡るため、そこはバグと判断する。

### 2. 両者0個公開時の -1 点は正式ルールか

正式ルールなら `fafnir_rule.txt` に追記する。
正式ルールでないなら `server_0424.py` の減点処理を削除する。

## 修正優先度案

1. プレイヤーへ相手の手札・未公開 bid を送らないようにする。
2. bid 提出を同時公開方式にする、または少なくとも両者提出完了まで相手へ内容を送らない。
3. 得点計算時の5個以上ルールを修正する。
4. ラウンド終了時の得点計算対象手札を `round_end_info` に保存し、確認後に次ラウンドを配る。
5. 両者0個時の -1 点を仕様として明記するか、実装から削除する。
6. `SCORE_40` を実際の勝利条件に合う文字列へ修正する。
